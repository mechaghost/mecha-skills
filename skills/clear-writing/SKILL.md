# Clear Writing

Use this when text quality matters. The goal is readable, direct, personable, actionable writing with good spacing and information rhythm.

## Core Workflow

1. Identify the reader, purpose, and action the text should support.
2. Choose the shortest structure that does the job.
3. Draft in plain language with concrete nouns and active verbs.
4. Trim filler, repetition, hedging, throat-clearing, and padded summaries.
5. Check spacing, scanability, and rhythm before final output.

## Voice

- Readable before clever.
- Concise before comprehensive.
- Useful before decorative.
- Personable without being chatty.
- Direct without sounding cold.
- Confident without hype.

Avoid inflated claims, generic AI phrasing, false warmth, vague value props, and clever lines that slow comprehension.

## Shape

- Lead with the answer, decision, or next useful action.
- Use short paragraphs, usually 1-3 sentences.
- Break up dense sections. No walls of text.
- Use bullets or steps when they improve scanning.
- Keep headings short and functional.
- Put context after the main point, not before it.
- Stop cleanly instead of circling back.

## Editing Pass

Ask:

- Can the first sentence stand on its own?
- Is every paragraph doing a distinct job?
- Can a long sentence become two cleaner ones?
- Can an abstract claim become a concrete action, result, or example?
- Can the reader understand what to do next within a few seconds?

For final answers, prefer: what changed, what was verified, and any remaining gap or risk. Do not recap every step unless the user asks.

## Product And UI Copy

- Name the thing plainly.
- Button labels should use specific action verbs.
- Helper text should reduce uncertainty, not explain obvious UI.
- Error text should say what happened and how to recover.
- Empty states should point to the next useful action.
- Marketing copy should be specific about audience, offer, proof, and action.
- For UI implementation details, pair this with `web-design`.

## Typography

- When choosing a readable body or UI font and the project has no established typography system, prefer Lexend.
- Reuse existing font tokens, theme files, and shared text components before adding one-offs.
- Do not force Lexend into plain text, code comments, terminal output, or a mature design system without a reason.
